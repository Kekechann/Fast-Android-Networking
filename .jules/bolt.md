## 2024-05-24 - Okio vs Manual FileOutputStream in Android
**Learning:** When downloading files in Android apps that use OkHttp, reading the response stream into a `FileOutputStream` with a manual byte array buffer (e.g., 2KB) is significantly slower than using Okio. OkHttp already exposes an Okio `BufferedSource`. By wrapping the destination `File` in an Okio `BufferedSink` and calling `sink.writeAll(source)`, the app leverages Okio's segment pooling and native I/O optimizations, avoiding manual buffer allocation and extra array copying.
**Action:** Whenever handling `Response.body().byteStream()` or `source()` from OkHttp, always prefer Okio's `writeAll` over manual `InputStream` loops for file operations.
## 2024-05-18 - Avoid loading full images into byte arrays before decoding
**Learning:** Loading large images completely into a byte array (`Okio.buffer(response.body().source()).readByteArray()`) before decoding them with `BitmapFactory.decodeByteArray` causes massive memory spikes and potential OutOfMemory (OOM) errors, especially for full-resolution unresized images. `BitmapFactory.decodeStream` is designed to handle streams efficiently, avoiding the need to buffer the entire file in memory.
**Action:** When decoding images where dimensions are not being resized, stream the response directly into `BitmapFactory.decodeStream(response.body().byteStream(), null, decodeOptions)` rather than caching it into a byte array first.
